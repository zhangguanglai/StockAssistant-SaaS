
#### 🚨 发现问题1：总分上限超出PRD声明

**根因：** `trend_score = int(trend_score * 35 / 40) + hl_score`（L2887）

```python
# 最大涨幅评分: 8<=pct<=9.5 → +20, 最大换手: 8<=turn<=12 → +20
# → trend_score_raw = 40
# → int(40 * 35 / 40) = int(35.0) = 35 ✓
# → + hl_score(max=10) → trend_score_max = 45
# → 总分 = 45 + 25 + 20 + 15 = 105分（非PRD声明的100分）
```

**影响：** 策略总分理论最高105分，与PRD声明的"满分100分"不符（差5分）

**修复建议：**
```python
# 方案1：修复 trend_score 转换（去掉整数截断）
trend_score = round(trend_score_raw * 35 / 40) + hl_score
# 40 * 35/40 = 35, 35 * 35/40 = 30.625 → round = 31
# 总分 = 31 + 10 + 25 + 20 + 15 = 101分（仍然超过）

# 方案2（推荐）：在总分上加强制 cap
total_score = min(trend_score + sector_score + money_score + bonus_score, 100)
```

#### 🚨 发现问题2：涨换评分条件区间边界

**PRD描述：**
```
涨幅评分：8%以下=8分 / 6-8%=15分 / 8%以上=20分
```

**代码实现（L2776-2788）：**
```python
if 8 <= pct_chg <= 9.5:  # 8-9.5% → 20分 ✓
    trend_score += 20
elif 6 <= pct_chg < 8:    # 6-8% → 15分 ✓
    trend_score += 15
else:                     # <6% → 8分
    trend_score += 8
```

⚠️ **PRD描述"8%以下=8分"包含<6%区间，与代码一致（<6%得8分）**
⚠️ **PRD描述"8%以上=20分"包含>9.5%区间（涨停股），但策略设计限定5-9.5%，超出部分只拿到20分而非最高**

#### 审计轨迹核查 ✅

`sector_leader` 策略在 `run_sector_leader_screener()` 内部生成 `match_audit_sl`（L2895-2979）：
- `gates`（6个门禁）：板块涨幅 / 个股涨幅范围 / 换手率范围 / 主力净流入 / 市值范围 / 三看确认
- `scoring`（4维度）：趋势面(35) / 板块面(25) / 资金面(20) / 加分项(15)
- `metrics`：pct_chg / turnover_rate / circ_mv_yi / total_net_in / inflow_days / hl_structure / three_views_count

## 三、共性问题与风险

### 3.1 🚨 高优先级：板块龙头总分超出PRD声明

**问题：** sector_leader 理论最高105分，PRD声明满分100分
**影响：** 评分超出100分会让用户困惑，且与其他两个策略的满分标准不统一
**修复建议：** 在 `run_sector_leader_screener()` 的 `passed.append()` 之前添加：
```python
total_score = min(total_score, 100)  # 确保不超过100分
```

### 3.2 ⚠️ 中优先级：数据源T+1延迟需显式告知用户

三个策略均重度依赖 Tushare 数据（ths_daily / moneyflow），这些数据有T+1延迟。

**建议：** 在选股结果页面或策略卡片中增加数据时效性标注：
```
⚠️ 板块/资金数据为T+1（昨日收盘后更新），仅供参考
```

### 3.3 ⚠️ 中优先级：审计轨迹实现位置不统一

| 策略 | 审计轨迹生成位置 | 一致性 |
|------|----------------|--------|
| trend_break | `final_ranking()` 统一生成 | ✅ 统一 |
| sector_leader | 各函数内部生成 | ⚠️ 分散 |
| oversold_bounce | 各函数内部生成 | ⚠️ 分散 |

**建议：** 统一在 `run_strategy()` 出口处统一生成 `match_audit`，便于维护

### 3.4 ℹ️ 低优先级：三看确认在板块龙头中描述歧义

**现状：** sector_leader 的 gates 中写">=2/3(加分项)"，但三看是加分不是 gate
**建议：** 改为更准确的标签，如"三看确认(加分项: 3/3=10分, 2/3=5分, 1/3=2分)"


**建议：** 修改 `save_screen_history` 保存完整 `results`（含 `match_audit`），而不只是 `top5` 摘要。
