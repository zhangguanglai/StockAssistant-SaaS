## 二、P0 致命问题（必须立即修复）

### S-01: Tushare Token 和 SECRET_KEY 硬编码在代码中
- **文件**: `config.py` 第5行、第26行
- **问题**: Token `e424926a...` 和 JWT密钥 `a-stock-portfolio-saas-secret-2026` 直接写死在源码中，代码已提交到仓库
- **风险**: 密钥泄露 → 任何人可伪造 JWT token、盗用 Tushare 配额
- **修复**:
  ```python
  # config.py — 改为环境变量读取
  import os
  TUSHARE_TOKEN = os.environ.get("TUSHARE_TOKEN", "")
  SECRET_KEY = os.environ.get("SECRET_KEY", secrets.token_hex(32))
  ```

### S-02: JWT 未校验 alg 字段（算法混淆攻击）
- **文件**: `auth.py` 第76-98行
- **问题**: `_jwt_decode()` 未验证 header 中的 `alg` 是否为 `HS256`，攻击者可将 header 改为 `{"alg":"none"}` 绕过签名验证
- **风险**: 认证绕过 → 任意用户身份冒充
- **修复**: 在第84行后增加
  ```python
  header = json.loads(_base64url_decode(header_b64))
  if header.get("alg") != "HS256":
      return None
  ```

### S-03: SQL 注入风险
- **文件**: `database.py` 第48行、`server.py` 多处 f-string SQL
- **问题**: `_get_table_columns(cur, table_name)` 直接拼接表名到 SQL 中。虽然此处 `table_name` 来自内部调用（安全），但属于危险编码模式
- **更严重的隐患**: `server.py` 中 `update_profile` (531行) 虽然使用了参数化查询，但项目缺乏统一的 SQL 安全审查流程，后续开发者容易复制 f-string 模式
- **修复**: 对所有 f-string SQL 进行审计，确保使用参数化查询。`_get_table_columns` 中的表名应做白名单校验
  ```python
  ALLOWED_TABLES = {"users", "positions", "trades", "capital", "watch_list"}
  if table_name not in ALLOWED_TABLES:
      raise ValueError(f"Invalid table name: {table_name}")
  ```

### S-04: PBKDF2 迭代次数不足
- **文件**: `auth.py` 第27行
- **问题**: `hashlib.pbkdf2_hmac('sha256', ..., 100000)` 使用 100,000 次迭代
- **风险**: OWASP 2023 推荐 310,000+ 次迭代（SHA-256）。当前值使暴力破解成本偏低
- **修复**:
  ```python
  PBKDF2_ITERATIONS = 310000  # OWASP 2023 推荐
  dk = hashlib.pbkdf2_hmac('sha256', password.encode('utf-8'), salt.encode('utf-8'), PBKDF2_ITERATIONS)
  ```
  > 注意：需要考虑已有用户的密码迁移策略

### D-01: 买入操作无现金余额校验 → 允许负现金
- **文件**: `server.py` 第767-771行 (add_position)、第864-868行 (add_trade)
- **问题**: 买入时直接 `cash -= buy_amount`，未检查余额是否充足
- **风险**: 现金可为负数 → 总资产虚高 → 用户看到不真实的收益数据
- **影响范围**: 直接影响"资产总览"面板的所有数据准确性
- **修复**:
  ```python
  buy_amount = buy_price * buy_volume + fee
  current_cash = data["capital"].get("cash", 0)
  if current_cash < buy_amount:
      return jsonify({"error": f"可用现金不足，需要 ¥{buy_amount:.2f}，当前仅 ¥{current_cash:.2f}"}), 400
  data["capital"]["cash"] = round(current_cash - buy_amount, 2)
  ```

### D-02: 卖出未校验可卖数量
- **文件**: `server.py` 第876-896行 (sell_position)
- **问题**: `sell_volume > current_volume` 时仅返回提示但未阻止实际操作
- **修复**: 在执行卖出逻辑前严格校验
  ```python
  if sell_volume > current_volume:
      return jsonify({"error": f"卖出数量不能超过持仓 {current_volume} 股"}), 400
  if sell_volume <= 0 or sell_price <= 0:
      return jsonify({"error": "卖出价格和数量必须大于0"}), 400
  ```

### B-01: 并发请求导致数据覆盖
- **文件**: `server.py` load_portfolio/save_portfolio + `database.py` JSON 文件读写
- **问题**: 多个请求同时读写 `portfolio.json` 时，后写入的会覆盖先写入的数据（Last Write Wins）
- **风险场景**: 用户在手机端和电脑端同时操作，或前端快速连续点击买入/卖出
- **修复**: 
  - 短期：为 JSON 操作加文件锁（`fcntl.flock` / `msvcrt.locking`）
  - 长期：已完成 SQLite 迁移的表（positions/trades/capital）不再使用 JSON，消除此问题

### B-02: screen_history 仍使用 JSON 文件持久化
- **文件**: `screener.py`
- **问题**: 其他数据已迁移到 SQLite，但 screen_history 仍使用 `data/cache/screen_history.json`
- **风险**: 与 B-01 相同的并发覆盖风险 + SQLite 与 JSON 双写的一致性问题
- **修复**: 创建 `screen_history` 表，与其他业务数据统一使用 SQLite

## 七、结论

系统在**功能完整性**上表现优秀——选股引擎的三策略架构、策略效果跟踪闭环、5维持仓评分体系都属于高质量的产品设计。止损止盈的交互设计（支撑压力位点击填入）和情绪标签系统是亮点功能。

但系统存在**两类核心风险**需要优先处理：

1. **安全风险**：密钥硬编码、JWT 算法未校验、PBKDF2 强度不足——这些在多用户 SaaS 场景下是致命漏洞
2. **数据完整性风险**：买入无余额校验、并发数据覆盖——直接影响用户看到的资产数据是否可信

建议按照上述路线图，先完成第一、二阶段的修复（约2-3天工作量），确保系统安全性和数据可信度后，再进行前端体验和架构优化。
